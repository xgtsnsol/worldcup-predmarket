# Decentralized Prediction Market DApp – Execution Plan

## Goal
Build a web3 application on Solana that ingests real‑time TxLINE sports data, lets users place predictions in USDC escrow, and resolves outcomes via a trustless CPI into TxLINE’s `validate_stat` instruction. The app includes an Automated Market Maker (AMM) for liquidity, a keeper bot for settlement, and a clean UI built with Next.js (App Router) and Tailwind.

---

## Architecture Overview

````mermaid
flowchart LR
    subgraph Frontend
        UI[React UI] --> API[Next.js API Routes]
    end
    subgraph Backend
        API --> SolanaKit[Solana Kit Client]
        SolanaKit --> CPI[TxLINE CPI (validate_stat)]
        CPI --> TxLINE[TxLINE SSE Stream]
    end
    subgraph On‑Chain
        AMM[AMM Contract] --> Escrow[USDC Escrow PDA]
        Escrow --> Winners[Distribute Funds]
    end
    Keeper[Keeper Bot] --> CPI
````

- **Frontend**: Next.js 16 with App Router, Tailwind CSS, React hooks, and @solana/wallet-adapter for wallet connection.
- **Backend**: TypeScript client generated by **Codama** from the TxLINE Anchor IDL; used via `@solana/kit`.
- **On‑Chain**: Two Solana programs:
  1. **AMM/Order‑book contract** handling liquidity pools, price curves, and escrow.
  2. **Settlement contract** that CPI‑calls TxLINE’s `validate_stat` to verify match results.
- **Keeper**: A lightweight Node script (run with `bun`) that listens to the TxLINE SSE feed, detects match finalization, and triggers the settlement CPI.

## Milestones & Tasks

- **M1 – Project Scaffold** (done)
  - Next.js 16 with App Router, Tailwind, Bun as package manager.
  - Installed Tailwind, PostCSS, Autoprefixer.
  - Added `postcss.config.cjs`.

- **M2 – TxLINE Client Generation**
  1. Place TxLINE IDL (`txline.json`) in `idl/`.
  2. Run `bun run scripts/generate-client.ts` (Codama) → `src/txline-client/`.
  3. Export a thin wrapper using `@solana/kit` utilities.

- **M3 – On‑Chain Contracts**
  1. **AMM Contract** (`amm_program`)
     - PDA for pool state, USDC vault.
     - Swap function implementing constant‑product formula.
  2. **Settlement Contract** (`settlement_program`)
     - Store prediction bets (PDA per market).
     - `settle(market_id)` CPI‑calls TxLINE `validate_stat`.
  3. Write tests with `solana-test-validator`.

- **M4 – Frontend UI Components**
  - `Header` (wallet connect, theme toggle).
  - `MarketList` – lists active prediction markets, odds from TxLINE.
  - `MarketDetail` – place prediction, view liquidity, AMM price curve.
  - `LiveFeed` – SSE component showing real‑time match events.
  - `Admin/Keeper` UI (optional) to manually trigger settlement.

- **M5 – Keeper Bot**
  - Node script (`scripts/keeper.ts`).
  - Subscribes to TxLINE SSE, detects `status: "final"`.
  - Calls on‑chain `settle` via `@solana/kit`.

- **M6 – Deployment**
  - Vercel deployment for Next.js frontend.
  - Deploy programs to Solana **devnet** using `solana program deploy`.
  - Configure environment variables (`NEXT_PUBLIC_SOLANA_RPC`, `TXLINE_WS_URL`).

- **M7 – Demo & Documentation**
  - Write `README.md` with repo link, deployment steps, and TxLINE endpoints used.
  - Prepare a short feedback section about TxLINE integration experience.
  - Add comprehensive test suite covering unit, integration, and end‑to‑end scenarios.
